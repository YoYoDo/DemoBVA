# /speckit.specify - 功能規格建立

## 用途

從自然語言功能描述建立完整的功能規格書（spec.md）。

## 憲章要求 ⚠️

依據**原則九：訪談先行制度**，在開始撰寫規格前「必須」先完成使用者訪談。

## 執行步驟

### 步驟 0: 訪談先行（必要條件）🎯

**⚠️ 此步驟不可跳過**

在開始任何規格撰寫前，必須先進行結構化訪談：

1. **發起訪談**：向使用者提出以下核心問題：

   **Q1. 業務目標**
   - 此模組的主要業務目標是什麼？
   - 預期達成什麼成果？

   **Q2. 使用者角色**
   - 主要使用者是誰？
   - 他們的關鍵需求和痛點是什麼？

   **Q3. 整合關係**
   - 此模組需要與哪些現有模組或外部系統整合？
   - 有什麼資料交換需求？

   **Q4. 業務規則**
   - 有哪些關鍵業務規則或約束條件？
   - 有什麼必須遵守的法規或合規要求？

   **Q5. 優先級與範圍**
   - 功能的優先級如何排序？
   - 是否有階段性交付期望？
   - 什麼明確不在範圍內？

2. **記錄訪談內容**：將所有問答記錄下來

3. **確認理解**：將理解摘要回饋給使用者確認，確保無誤解

4. **獲得確認後**：才能進入步驟 1 開始規格撰寫

### 步驟 1: 生成分支名稱

- 分析功能描述，提取最有意義的關鍵詞
- 建立 2-4 個字詞的簡短名稱（例：user-auth、oauth2-api-integration）
- 使用動作-名詞格式

### 步驟 2: 檢查現有分支

```bash
# turbo
git fetch --all --prune
```

查找所有來源的最高特性編號：

- 遠端分支：`git ls-remote --heads origin | grep -E 'refs/heads/[0-9]+-'`
- 本地分支：`git branch | grep -E '^[* ]*[0-9]+-'`
- Specs 目錄：`ls specs/ | grep -E '^[0-9]+-'`

### 步驟 3: 建立新功能分支

```bash
.specify/scripts/bash/create-new-feature.sh --json --number <N+1> --short-name "<short-name>" "<功能描述>"
```

### 步驟 4: 載入規格模板

讀取 `.specify/templates/spec-template.md` 了解必需章節。

### 步驟 5: 執行規格工作流程

1. 解析使用者描述與訪談記錄，識別：角色、動作、資料、約束
2. 對於不清楚的方面：
   - 基於上下文和行業標準做出合理推測
   - 僅在顯著影響功能範圍時標記 `[NEEDS CLARIFICATION]`
   - **限制：最多 3 個 [NEEDS CLARIFICATION] 標記**
3. 填寫訪談記錄章節
4. 填寫使用者場景與測試章節
5. 生成功能需求（每個需求必須可測試）
6. 定義成功標準（技術無關、可衡量）
7. 識別關鍵實體

### 步驟 6: 寫入規格檔案

將規格寫入 `specs/<branch-name>/spec.md`，包含：

- **訪談記錄**：完整記錄步驟 0 的問答內容
- **使用者場景與測試**
- **需求**
- **成功標準**

### 步驟 7: 規格品質驗證

建立 `FEATURE_DIR/checklists/requirements.md` 檢查清單，驗證：

- 訪談記錄已包含
- 無實作細節（語言、框架、API）
- 專注於使用者價值和業務需求
- 所有強制章節已完成
- 需求可測試且無歧義
- 成功標準可衡量

### 步驟 8: 報告完成

輸出分支名稱、規格檔案路徑、檢查清單結果

## 一般準則

- 專注於使用者需要**什麼**和**為什麼**
- 避免**如何**實作（不包含技術堆疊、API、程式碼結構）
- 為業務利害關係人撰寫，非開發人員

## 訪談問題模板

可根據模組類型調整以下問題：

| 問題類別 | 核心問題 |
|----------|----------|
| 業務目標 | 此模組要解決什麼問題？預期成果是什麼？ |
| 使用者 | 主要使用者是誰？他們的關鍵需求？ |
| 整合 | 需要與哪些系統整合？資料如何流動？ |
| 規則 | 有哪些業務規則或合規要求？ |
| 優先級 | 功能優先級？階段性交付期望？ |
| 範圍 | 什麼明確不在範圍內？ |
