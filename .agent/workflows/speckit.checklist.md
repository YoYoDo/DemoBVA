# /speckit.checklist - 檢查清單生成

## 用途

為當前功能生成自訂檢查清單，用於驗證需求品質。

## 核心概念

**檢查清單是「需求的單元測試」** - 驗證需求的品質、清晰度和完整性。

**不是用於驗證/測試實作**:

- ❌ 不是「驗證按鈕點擊正確」
- ❌ 不是「測試錯誤處理正常工作」
- ❌ 不是「確認 API 返回 200」

**是用於需求品質驗證**:

- ✅ 「是否為所有卡片類型定義了視覺層次需求？」[完整性]
- ✅ 「'顯眼展示'是否有具體尺寸/定位量化？」[清晰度]
- ✅ 「所有互動元素的懸停狀態需求是否一致？」[一致性]

## 執行步驟

### 1. 設定環境

```bash
# turbo
.specify/scripts/bash/check-prerequisites.sh --json
```

### 2. 澄清意圖

生成最多 3 個上下文澄清問題：

- 範圍細化（例如：「是否應包含與 X 和 Y 的整合接觸點？」）
- 風險優先級（例如：「哪些潛在風險區域應有強制性閘門檢查？」）
- 深度校準（例如：「這是輕量級的預提交健康檢查還是正式發布閘門？」）

### 3. 理解使用者請求

結合使用者輸入與澄清答案：

- 推導檢查清單主題（如：security, review, deploy, ux）
- 整合使用者提及的必備項目
- 從 spec/plan/tasks 推斷缺失上下文

### 4. 載入功能上下文

從 FEATURE_DIR 讀取：

- spec.md: 功能需求和範圍
- plan.md（如存在）: 技術細節、依賴
- tasks.md（如存在）: 實作任務

### 5. 生成檢查清單

建立 `FEATURE_DIR/checklists/` 目錄（如不存在）

**品質維度分類**:

- **需求完整性** - 是否記錄了所有必要需求？
- **需求清晰度** - 需求是否具體且無歧義？
- **需求一致性** - 需求是否一致無衝突？
- **驗收標準品質** - 成功標準是否可衡量？
- **場景覆蓋** - 是否涵蓋所有流程/案例？
- **邊緣案例覆蓋** - 是否定義了邊界條件？

**項目結構**:
每個項目應遵循此模式：

- 問題格式詢問需求品質
- 專注於規格中**寫了什麼**（或沒寫什麼）
- 包含品質維度標籤 [完整性/清晰度/一致性等]
- 引用規格章節 `[Spec §X.Y]`
- 對於缺失需求使用 `[Gap]` 標記

**範例**:

```markdown
- [ ] CHK001 是否為所有 API 失敗模式定義了錯誤處理需求？[Gap]
- [ ] CHK002 '快速載入'是否有具體時間閾值量化？[清晰度, Spec §NFR-2]
- [ ] CHK003 導航需求是否在所有頁面間一致？[一致性, Spec §FR-10]
```

**🚫 絕對禁止**:

- ❌ 以「驗證」、「測試」、「確認」開頭 + 實作行為
- ❌ 引用程式碼執行、使用者動作、系統行為
- ❌ 「正確顯示」、「正常工作」、「如預期運作」

**✅ 必需模式**:

- ✅ 「是否為 [場景] 定義/指定/記錄了 [需求類型]？」
- ✅ 「[模糊術語] 是否有具體標準量化/澄清？」
- ✅ 「[章節 A] 和 [章節 B] 之間的需求是否一致？」

### 6. 報告

輸出檢查清單完整路徑、項目數，提醒每次執行建立新檔案。

## 檢查清單類型範例

- **UX 需求品質**: `ux.md`
- **API 需求品質**: `api.md`
- **效能需求品質**: `performance.md`
- **安全需求品質**: `security.md`
