# /speckit.clarify - 規格澄清

## 用途

偵測並減少活動功能規格中的歧義或缺失決策點，並將澄清內容直接記錄在規格檔案中。

## 注意

此澄清工作流程預期在調用 `/speckit.plan` **之前**執行（並完成）。

## 執行步驟

### 1. 設定環境

```bash
# turbo
.specify/scripts/bash/check-prerequisites.sh --json --paths-only
```

解析 JSON 取得：FEATURE_DIR, FEATURE_SPEC

### 2. 載入並掃描規格

載入當前規格檔案，使用以下分類法執行結構化歧義與覆蓋掃描。對每個類別標記狀態：Clear / Partial / Missing。

**功能範圍與行為**:

- 核心使用者目標與成功標準
- 明確的範圍外聲明
- 使用者角色/人物區分

**領域與資料模型**:

- 實體、屬性、關係
- 身份與唯一性規則
- 生命週期/狀態轉換
- 資料量/規模假設

**互動與 UX 流程**:

- 關鍵使用者旅程/序列
- 錯誤/空/載入狀態
- 無障礙或本地化備註

**非功能品質屬性**:

- 效能（延遲、吞吐量目標）
- 可擴展性（水平/垂直、限制）
- 可靠性與可用性（運行時間、恢復預期）
- 可觀測性（日誌、指標、追蹤信號）
- 安全與隱私（認證/授權、資料保護、威脅假設）

**整合與外部依賴**:

- 外部服務/API 及失敗模式
- 資料匯入/匯出格式
- 協議/版本假設

**邊緣案例與失敗處理**:

- 負面場景
- 速率限制/節流
- 衝突解決（如並發編輯）

### 3. 生成優先級問題佇列

（內部）生成候選澄清問題的優先級佇列（最多 5 個）。

約束：

- 整個會話總共最多 10 個問題
- 每個問題必須可用以下方式回答：
  - 簡短多選（2-5 個選項），或
  - 單詞/短語答案（「用 ≤5 個字回答」）
- 僅包含答案會實質影響架構、資料建模、任務分解、測試設計的問題

### 4. 順序提問迴圈（互動式）

- 一次呈現**恰好一個**問題
- 對於多選問題：
  - 分析所有選項，確定**最合適選項**
  - 在頂部突出呈現**推薦選項**並說明理由
  - 格式：`**推薦:** 選項 [X] - <理由>`
  - 然後以 Markdown 表格呈現所有選項
- 使用者回答後：
  - 如果回覆「yes」、「recommended」或「suggested」，使用先前陳述的推薦
  - 驗證答案對應到選項或符合 ≤5 字約束
  - 記錄到工作記憶中，進入下一個問題

**停止條件**:

- 所有關鍵歧義提前解決
- 使用者發出完成信號（「done」、「good」、「no more」）
- 達到 5 個已問問題

### 5. 每個接受答案後整合

- 確保存在 `## Clarifications` 章節
- 在其下建立 `### Session YYYY-MM-DD` 子標題
- 追加項目：`- Q: <問題> → A: <最終答案>`
- 立即將澄清應用到最適當的章節

### 6. 驗證

- 澄清會話每個接受答案恰好一個項目
- 總問題 ≤ 5
- 更新章節不包含新答案應解決的殘留模糊佔位符
- Markdown 結構有效

### 7. 寫回規格

將更新的規格寫回 `FEATURE_SPEC`

### 8. 報告完成

- 問題數與已回答數
- 更新規格路徑
- 觸及章節（列出名稱）
- 覆蓋摘要表
- 建議下一個命令

## 行為規則

- 如果未發現有意義歧義：「未偵測到值得正式澄清的關鍵歧義。」
- 如果規格檔案缺失：指示使用者先執行 `/speckit.specify`
- 永不超過 5 個已問問題
- 尊重使用者提前終止信號
